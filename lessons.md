# FX Predict 项目经验教训

## 开发经验

### 1. 数据泄漏防范
- 所有技术指标和特征计算必须使用 `.shift(1)` 使用滞后数据
- 在测试中验证至少 70% 的指标在第一行是 NaN
- 目标变量使用 `shift(-horizon)` 仅用于训练阶段，预测时不可用

### 2. 硬性约束的重要性
- ML probability ≤ 0.50 时绝对禁止买入
- ML probability ≥ 0.60 且预测为 1 时推荐买入
- 在集成测试中验证硬性约束是否生效

### 3. 配置管理的模块化
- 使用 `config.py` 集中管理所有配置
- 使用环境变量（.env）存储敏感信息（API 密钥）
- 实现配置验证函数
- 使用类型提示提高代码可读性

### 4. 数据文件路径的三级配置
- 命令行参数（最高优先级）
- 环境变量（中等优先级）
- 配置文件默认值（最低优先级）

### 5. 测试驱动的开发（TDD）
- 先写测试，再写实现代码
- 确保代码质量
- 提前发现问题
- 便于重构

### 6. 模型持久化
- 使用 pickle 序列化模型
- 为每个货币对创建独立的模型文件
- 文件命名规范：`{pair}_{model_name}.pkl`

### 7. 错误处理和日志记录
- 所有 API 调用都有异常处理
- 使用 logging 模块记录日志
- 区分日志级别（INFO、WARNING、ERROR）

### 8. 集成测试的重要性
- 完整工作流：数据加载 → 技术指标 → 特征工程 → 模型训练 → 预测 → 综合分析
- 多货币对测试
- 硬性约束验证
- 数据泄漏防范验证

### 9. 技术指标的选择
- 基于金融理论
- 经过广泛验证
- 覆盖不同类别（趋势、动量、波动、成交量）

### 10. 特征工程的平衡
- 选择相关性强的特征
- 避免多重共线性
- 使用滞后特征防止数据泄漏

### 11. Python 模块的主函数入口
- 使用 `python3 -m module_name` 方式运行
- 在主函数中使用 `argparse` 解析命令行参数

### 12. Git 工作流
- 频繁提交和小步提交
- 提交信息清晰规范（使用 Conventional Commits）
- 提交前运行测试确保通过

### 13. 前端状态管理的重要性
- 复杂的交互需要维护全局状态
- 根据触发源执行不同的逻辑
- 标签页切换时获取对应数据

### 14. 前端防御性编程
- 使用可选链操作符 `?.` 避免访问 undefined
- 提供默认值确保代码健壮性
- 验证必需字段是否存在

### 15. 数据结构设计的重要性
- 不同的 API 端点可能需要不同的数据结构
- 标准化数据适合前端展示
- 原始数据适合需要详细信息的功能

### 16. HTML 结构对 CSS 布局的影响
- 标签页应该放在标题容器外部
- 避免与标题和关闭按钮混在一起
- DOM 结构直接影响 CSS 布局

### 17. 多轮修复的必要性
- 复杂功能通常需要多轮迭代
- 每轮修复解决特定问题
- 用户反馈驱动优化

### 18. 设计先行的重要性
- 详细的设计文档减少后期返工
- 实现计划明确任务步骤
- 逐个完成任务验证功能

### 19. 浏览器缓存和硬刷新
- 前端静态资源更新后需要硬刷新
- 提醒用户使用 Ctrl+Shift+R
- 或者手动清除缓存

### 20. 后端缓存陷阱
- 数据结构更新后必须重启服务器
- 缓存机制可能导致数据不一致
- 及时清除缓存确保最新数据

### 21. cron 环境的特殊性
- cron 默认 PATH 很有限，需要手动设置
- Python 模块路径需要通过 PYTHONPATH 显式指定
- 在 crontab 文件中显式设置环境变量

### 22. Git 仓库管理 - 二进制文件处理
- 生成的文件不应该提交到 Git
- 使用 .gitignore 排除临时文件、缓存文件、日志文件
- 模型文件应该在训练时生成

### 23. Docker 镜像的独立性
- 镜像应该包含所有必要的组件
- 环境变量应该正确设置
- cron 任务应该能够独立运行

### 24. 技能系统的设计原则
- 独立性：技能应该独立且可复用
- 跨平台：支持多种操作系统
- 文档完整：提供清晰的使用文档和示例
- 标准接口：使用标准的命令行参数和输出格式

## 代码质量问题

### 1. 代码修改后必须进行语法检查
- 使用 Node.js 的语法检查工具：`node -c server.js`
- 在每次修改后立即运行语法检查
- 确保没有语法错误后再启动服务器

### 2. 测试覆盖率不均
- 部分模块测试覆盖率低于目标（70%）
- 需要添加更多特征验证测试
- 实现大模型模块的 mock 测试

## 性能优化建议

### 1. 缓存机制
- 缓存技术指标计算结果，避免重复计算
- 使用 `@lru_cache(maxsize=128)` 装饰器

### 2. 并行计算
- 使用多进程并行计算多个货币对的预测

### 3. 数据预处理优化
- 使用更高效的数据处理库（如 Polars）

## 部署考虑

### 1. 容器化
- 使用 Docker 容器化应用
- 提供完整的 Dockerfile 和 docker-compose.yml

### 2. 环境变量管理
- 使用 python-dotenv 加载 .env 文件
- 敏感信息不提交到 Git

### 3. 日志管理
- 使用日志轮转和集中管理
- 使用 `RotatingFileHandler`

## 用户反馈和改进

### 1. 命令行界面
- 提供更友好的 CLI 接口

### 2. 可视化
- 添加图表可视化功能

### 3. 实时更新
- 支持实时数据更新和预测

5. **文档**：完善的文档是项目成功的基础

8. **数据结构设计**：根据使用场景设计合适的数据结构
9. **多轮修复**：复杂功能需要多轮迭代和用户反馈
10. **设计先行**：详细的设计文档可以减少后期返工
